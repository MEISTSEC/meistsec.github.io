Firewall Configurations

By Meistsec

April 19, 2022

4/19/2022:

So I decided to make the transition from Ubuntu 20.04 LTS to Fedora Workstation 36 Beta as my host operating system. Over the years, I have become incredibly frustrated with Ubuntu regarding the numerous changes implemented by Canonical (e.g., Snaps, Synaptic, etc. ). These changes have been troublesome because this is an operating system that I have been using since 2008 when I first started my return to studying information technology. Even though it is still very functional, and I love KDE Plasma, no doubt about it, I grudgingly do not feel it is a polished modern-day workstation for the normal Linux home enthusiast.

When I started using Ubuntu, I utilized IPTables for my firewall management system and later moved to The Uncomplicated Firewall (UFW) application. However, my transition to Fedora presented a problem because Fedora 36 natively supports Firewalld.

So, like so many other things with being a day-to-day Linux user, it was time to bust out the Fedora docs, forums, man pages, and “The Google.”

**FIREWALLD:
**Provides a dynamically managed firewall with support for network/firewall zones that define the trust level of network connections or interfaces. It supports IPv4, IPv6 firewall settings, Ethernet bridges, and IP sets. There is a separation of runtime and permanent configuration options. It also provides an interface for services or applications to add firewall rules directly. firewalld.org

Environment

With using Fedora as my host/base operating system, I also have to mention that I deploy numerous virtual machines for security research purposes. Over the past few years, I have been an avid Virtual-box user. However, I love the performance offered by Qemu, managed by the Virtmanager package provided within numerous Linux repositories. It is a far superior product in my opinion.

When I started exploring firewalld after my initial install of Fedora, I was concerned that the base installation left the working host environment vulnerable in its predefined firewall services. This security limitation is, by purpose, to enable the ease of use for a normal day-to-day user. However, not to scare recent converts, it’s necessary to mention these services like SSH are not running in systemd within the installation natively.

So, with that, there is no concern of a brute force attack on port 22 presented by default. Although, it is fair to mention that SSH and other numerous services are allowed organically within the firewalld rule set in the “FedoraWorkstation” zone in a normal Fedora install. So, this leaves the firewall vulnerable as an entry point if those services are remotely/locally made available.

One problem I did encounter in transitioning my research machine from Ubuntu to Fedora was the ability to host a Python HTTP server to transfer files from the host to my virtual machines using wget, curl, or certutil in Powershell.

With that, the following commands will show you how to configure firewalld to lock-down/harden your machine and allow you to transfer files to or from your host/virtual instances.

Cheers!

Commands

Execute all commands using “sudo” if you are not a root user.

==[root@FedoraWorkstation ~]# firewall-cmd --get-active-zones==

This command lists all active “zones” within your firewall

==[root@FedoraWorkstation ~]# firewall-cmd --zone=FedoraWorkstation --list-all==

This command lists all the active configurations within your public firewall configuration on your Ethernet connection. By default, the FedoraWorkstation profile authorizes trust to dhcpv6-client, mdns, samba-client, and ssh.

==[root@FedoraWorkstation ~]# firewall-cmd --zone=FedoraWorkstation --remove service=dhcpv6-client==

Dump those unnecessary services you do not need. Ensure you are testing the TCP/UDP protocols are you normally use as you remove services. Remember all you need to do is change [remove] to [add] in you command instructions.

==[root@FedoraWorkstation ~]# firewall-cmd --get-active-zones==

Check again after removing the necessary services from your rules list to ensure everything is correct.

==[root@FedoraWorkstation ~]# firewall-cmd --zone=libvirt --list-all==

Same thing. Remove unnecessary services from your virtualization profile.

Now its time to add rules to allow specific interfacing between your host and your virtual machines

==[root@FedoraWorkstation ~]# firewall-cmd --zone=FedoraWorkstation --add-source=192.168.x.x==

Do this for each virtual machine. This allowed me to server files using HTTP to my virtual instances.

Persistence

==[root@FedoraWorkstation ~]# firewall-cmd --runtime-to-permanent==

==[root@FedoraWorkstation ~]# systemctl reload firewalld==

==[root@FedoraWorkstation ~]# firewall-cmd --get-active-zones==

==[root@FedoraWorkstation ~]# firewall-cmd --zone=FedoraWorkstation --list-all==

==[root@FedoraWorkstation ~]# firewall-cmd --zone=libvirt --list-all==

Conclusion

This experience transitioning from UFW to Firewalld shows how Fedora/Redhat has focused their adopted software on efficient implementations within their curated catalog. It just makes sense from a security management perspective using a zone defense in managing multiple networking interfaces hosted from the user.
